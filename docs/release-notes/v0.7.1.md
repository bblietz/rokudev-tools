# rokudev-tools v0.7.1 brs-docs companion-lock fix

**Date:** 2026-05-20
**Type:** Patch release. Single bug fix on top of v0.7.0.

## Bug fix

**`brs-docs version` always reported "no corpus installed" against
dev / source installs.**

`build_corpus` was writing the companion lock as `corpus.sqlite.lock`
(via a misleading `out_path.with_suffix(".sqlite.lock")`), but
`first_run.py` and `cli.py` both look for `corpus.lock` beside the
sqlite. After first-run install, the cache contained `corpus.sqlite`
but no `corpus.lock`, so the version subcommand printed:

```
brs-docs 0.7.0
no corpus installed
telemetry: none
```

Wheel installs were unaffected (the wheel's `force-include` ships
`corpus.lock` at the right slot regardless of the build orchestrator).
Dev / editable / source installs hit the bug.

## Why the test suite didn't catch it

Two assertions were too loose:

- `tests/test_corpus_build/test_build.py` accepted EITHER
  `corpus.sqlite.lock` OR `corpus.lock` via an `or` clause.
- `tests/test_cli.py::test_version_subcommand` only asserted the
  package version string was in the output, never the corpus summary
  line.

Both tightened to fail fast on the regression.

## Changes

- `packages/brs-docs/src/brs_docs/corpus/build.py`: write companion
  lock as `corpus.lock` (sibling of `corpus.sqlite`).
- `packages/brs-docs/tests/test_corpus_build/test_build.py`: strict
  assertion that `corpus.lock` exists beside the sqlite.
- `packages/brs-docs/tests/test_cli.py`: assert `dev_doc=` is in
  `version` output and "no corpus installed" is NOT.
- `packages/brs-docs/.gitignore`: ignore the new artifact name.
- Version bump `__version__` + `pyproject.toml` + `corpus.lock`
  `brs_docs_version` 0.7.0 → 0.7.1.
