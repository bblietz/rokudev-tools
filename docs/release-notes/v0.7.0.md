# rokudev-tools v0.7.0 brs-docs MCP

**Date:** 2026-05-19
**Plan:** Plan 6 (brs-docs MCP)
**Spec:** `docs/superpowers/specs/2026-05-19-brs-docs-mcp-design.md`

## Headline

First release of `brs-docs`, a Python MCP server + CLI that gives
Claude Code (and a human dev) fast offline lookup over the Roku
BrightScript reference corpus, code samples, and rokudev-tools
feature modules + templates.

Part of the post-pivot dev-tools work order (block 1 of 9).

## What's new

- **5 MCP tools**: `brs_search`, `brs_get`, `brs_list`,
  `brs_sample_read`, `brs_recommend`.
- **SQLite + FTS5 corpus** (~11.4 MB; 1898 docs across 3 kinds at
  current upstream SHAs): sample (1889), template (7),
  feature_module (2). Schema supports all 9 kinds (component,
  interface, event, node, global_function, guide, sample,
  feature_module, template); the others are populated as upstream
  `rokudev/dev-doc` adds frontmatter'd markdown.
- **BM25 + tag-boost + module-category-boost ranker** (no embeddings,
  no LLM). v1 `tags.toml` ships keyword mappings + module categories
  + BM25 weights.
- **CLI**: `brs-docs serve | search | get | list | recommend |
  refresh | version`.
- **Transactional refresh** with atomic rename. Never corrupts the
  installed corpus on failure.
- **Production GitHub-fetch** in `build_corpus` (downloads tarballs
  at SHAs pinned in `corpus.lock`).
- **CI**: GitHub Actions Python job + corpus.lock + tags.toml
  discipline scripts.

## Scope cuts (deferred to v1.x)

- **No upstream-fetch refresh in CLI.** `brs-docs refresh` rebuilds
  from the bundled snapshot. True "fetch from GitHub at HEAD" needs
  network + auth + retries; T28 implemented the production fetch
  path inside `build_corpus`, but the `refresh` CLI command does
  NOT call it yet.
- **No corp-network mirror flag** (`internal.use_corp_doc_mirror`).
- **No `brs` umbrella CLI integration** (block 6 territory).
- **No LSP-as-tool composition** (block 2).
- **English-only corpus.**

## Test counts

- 92 default pytest tests passing.
- 19 integration tests (run via `pytest -m integration`):
  4 pass, 15 xfail (recommend fixture-pinned cases that reference
  paused v1 modules or guides not present in the tiny test fixture).

## Real-data quirk (T28)

`rokudev/dev-doc` @ `2fd52273e2c7a40cb358ee760f8070d55b44c948` is
essentially an empty repo (only `.gitignore`). The dev_doc scraper
produces 0 docs from upstream, but the FTS canary queries (`RowList`,
`ifString`) still pass because they're present in the samples corpus.
The scraper itself is unchanged; this is purely an upstream-content
gap, not a fetch-path bug.
