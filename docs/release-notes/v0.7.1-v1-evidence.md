## Bugs uncovered during verification

### Bug 1 (FIXED in 0be7237 + released as v0.7.1)

`build_corpus` wrote the companion lock as `corpus.sqlite.lock` while `first_run.py` + `cli.py` look for `corpus.lock`. Net effect: `brs-docs version` always reported "no corpus installed" against dev/source installs. Wheel installs were unaffected (force-include shipped the lock at the right slot). See `docs/release-notes/v0.7.1.md`.

### Bug 2 (NEW, ships in v0.7.0 and v0.7.1 wheels): wheel does NOT bundle corpus.sqlite

The v0.7.1 wheel built by `uv build` is **27 KB**, not the ~18 MB the spec targets. `unzip -l dist/brs_docs-0.7.1-py3-none-any.whl` shows `brs_docs/data/corpus.lock` (force-included) but no `brs_docs/data/corpus.sqlite`. Root cause:

1. `src/brs_docs/data/corpus.sqlite` is in `packages/brs-docs/.gitignore`.
2. Hatch's wheel target defaults to honoring `.gitignore`.
3. `pyproject.toml` `[tool.hatch.build.targets.wheel.force-include]` only force-includes `corpus.lock`, not `corpus.sqlite`.

Net effect: a wheel install would always hit `CorpusNotInitialized` on first run because the bundled sqlite is absent. `brs-docs version` would report `no corpus installed` and the CLI tools would error.

This bug was latent in v0.7.0 too. The dev-checkout `uv run` path works because `importlib.resources.files("brs_docs").joinpath("data/corpus.sqlite")` resolves to the live `src/brs_docs/data/corpus.sqlite`, which IS on disk after `make build-corpus`.

**Fix sketch (one-line pyproject.toml change):** add the sqlite to `force-include` (or use `[tool.hatch.build.targets.wheel] artifacts = ["src/brs_docs/data/corpus.sqlite"]`). Pair with a wheel-inspection test under `tests/contract/`. A v0.7.2 patch.

### Partial criterion #2: corpus has 3 kinds, not 9

`tests/contract/test_corpus_version_matches_brs_gen.py` + `test_build_includes_all_kinds` cover the 6 brs-gen-derived kinds (`component`, `node`, `global_function`, `sample`, `feature_module`, `template`) in the FIXTURE corpus. The bundled corpus built from upstream `rokudev/dev-doc` at SHA `2fd52273` exposes only 3 kinds at present:

- `sample`: 1889
- `template`: 7
- `feature_module`: 2

Spec §"Pre-bundled corpus contains at least 600 docs across all 9 kinds": **doc count met (1898 > 600), kind count NOT met (3 < 9)**. The 6 missing kinds (`component`, `interface`, `event`, `node`, `global_function`, `guide`) all depend on upstream `dev-doc` markdown gaining structured frontmatter that the scraper recognizes. Per spec §"What's new" footnote: "the others are populated as upstream `rokudev/dev-doc` adds frontmatter'd markdown". Acceptable v1 deficit per author intent, but worth tracking.
